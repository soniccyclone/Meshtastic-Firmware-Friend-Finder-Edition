Propose a new change — create the change, generate prose artifacts, and file the implementation tasks in **beads**.

I'll create:
- proposal.md (what & why)
- design.md (how)
- One beads issue per implementation step, labeled `openspec:<change-name>`

When ready to implement, run `/opsx:apply`.

---

**Conventions**

- Implementation tasks live in **beads**, not in a `tasks.md` file. Each task is a beads issue labeled `openspec:<change-name>`.
- The openspec `tasks` artifact is **not** generated by this skill. If the schema's `applyRequires` includes `tasks`, satisfy that requirement by writing a stub `tasks.md` that points at beads (see step 6).
- Use `bd` for tracking implementation progress through the change. Do **not** use TodoWrite or markdown checklists for task tracking.

**Input**: The user's request should include a change name (kebab-case) OR a description of what they want to build.

**Steps**

1. **If no clear input provided, ask what they want to build**

   Use the **AskUserQuestion tool** (open-ended, no preset options) to ask:
   > "What change do you want to work on? Describe what you want to build or fix."

   From their description, derive a kebab-case name (e.g., "add user authentication" → `add-user-auth`).

   **IMPORTANT**: Do NOT proceed without understanding what the user wants to build.

2. **Create the change directory**
   ```bash
   openspec new change "<name>"
   ```
   This creates a scaffolded change at `openspec/changes/<name>/` with `.openspec.yaml`.

3. **Sync beads and get artifact build order**
   ```bash
   bd dolt pull                                  # sync beads first
   openspec status --change "<name>" --json      # artifact dependency info
   ```
   Parse the openspec JSON to get:
   - `applyRequires`: array of artifact IDs needed before implementation (e.g., `["tasks"]`)
   - `artifacts`: list of all artifacts with their status and dependencies

4. **Ground the proposal in real source code (MANDATORY before writing any prose artifact)**

   Before generating proposal.md, design.md, or any spec, identify every external code surface the change will touch and **read it**. Specs that name struct fields, function signatures, file paths, or API symbols sourced from imagination produce normative requirements that turn out to be wrong at apply-time.

   a. **Resolve cross-repo source.** If the change touches code in another repository (e.g. an upstream firmware tree cloned by a Makefile, a vendored submodule, a sibling package), check whether that source is present locally:
      ```bash
      # Example: this project's Makefile clones LeapYeet/firmware into ../LeapYeet-firmware via 'make setup'.
      # Adapt to whatever the project's build/setup target is.
      ls "$EXTERNAL_SRC_PATH" 2>/dev/null || (cd .. && git clone --depth 1 --recurse-submodules "$EXTERNAL_REPO_URL" "$(basename "$EXTERNAL_SRC_PATH")")
      ```
      Read the project's Makefile / setup script first to find the canonical clone path and command — do not invent one.

   b. **Read the actual code.** Open the structs, functions, files, and APIs the change will modify. Capture verbatim:
      - Struct/class definitions with all fields and types
      - Function signatures and default-parameter values
      - File paths and conditional-compilation guards
      - Any platform-specific gating that affects the change

   c. **Verification rule for prose artifacts.** Every named struct field, function, file path, or API symbol that appears in proposal.md, design.md, or specs MUST trace back to a `grep`-verifiable line in real code. If a name truly cannot be verified before writing, mark it explicitly as `<TBD>` rather than asserting an invented name. Open Questions are NOT a substitute for verification — they are for genuine design decisions, not for "I didn't read the code yet."

   d. **Skip clause.** If the change is purely a docs/process/config tweak that touches no code surface, this step is a no-op — note that and continue.

5. **Create prose artifacts in dependency order (skip `tasks`)**

   Loop through artifacts whose dependencies are satisfied, **skipping the `tasks` artifact** (it's handled in step 6):

   a. **For each non-`tasks` artifact that is `ready`**:
      - Get instructions:
        ```bash
        openspec instructions <artifact-id> --change "<name>" --json
        ```
      - The instructions JSON includes:
        - `context`: Project background (constraints for you — do NOT include in output)
        - `rules`: Artifact-specific rules (constraints for you — do NOT include in output)
        - `template`: The structure to use for your output file
        - `instruction`: Schema-specific guidance for this artifact type
        - `outputPath`: Where to write the artifact
        - `dependencies`: Completed artifacts to read for context
      - Read any completed dependency files for context
      - Create the artifact file using `template` as the structure
      - Apply `context` and `rules` as constraints — but do NOT copy them into the file
      - Show brief progress: "Created <artifact-id>"

   b. **Continue until all non-`tasks` `applyRequires` artifacts are done**:
      - After creating each artifact, re-run `openspec status --change "<name>" --json`

   c. **If an artifact requires user input** (unclear context):
      - Use **AskUserQuestion tool** to clarify, then continue.

6. **File implementation tasks in beads**

   Read the proposal and design (and any specs you produced) and decompose the work into discrete, ordered implementation tasks. Then file each as a beads issue:

   ```bash
   bd create \
     --title="<short imperative task title>" \
     --description="<why + what + acceptance>" \
     --type=task \
     --priority=2 \
     --label="openspec:<name>"
   ```

   Guidelines for decomposition:
   - Each task should be independently implementable and reviewable.
   - Order matters: prefer 4–10 tasks for a typical change. Don't over-fragment.
   - For tasks that depend on other tasks, link them after creation:
     ```bash
     bd dep add <dependent-id> <prerequisite-id>
     ```
   - If a single broad task makes more sense as a parent epic, create the epic with `--type=feature` and file children, then link with `bd dep add` or use the project's epic conventions.

   When creating multiple issues, run `bd create` calls in **parallel** (multiple Bash tool calls in one message) for efficiency.

   **If the schema's `applyRequires` includes `tasks`** (most spec-driven schemas do): write a minimal stub at the artifact's `outputPath` so openspec considers the requirement satisfied:

   ```markdown
   # Tasks

   Implementation tasks for this change are tracked in **beads** under the label
   `openspec:<name>`.

   View them with:

       bd list --label="openspec:<name>"

   Run `/opsx:apply` to begin implementation.
   ```

   This keeps `openspec status` happy without duplicating the task list.

7. **Show final status**
   ```bash
   openspec status --change "<name>"
   bd list --label="openspec:<name>"
   ```

**Output**

After completing, summarize:
- Change name and location
- Prose artifacts created (proposal, design, specs as applicable)
- Number of beads tasks filed and their IDs
- "All artifacts created and tasks filed in beads. Ready for implementation."
- Prompt: "Run `/opsx:apply` or ask me to implement to start working on the tasks."

**Artifact Creation Guidelines**

- Follow the `instruction` field from `openspec instructions` for each prose artifact.
- The schema defines what each artifact should contain — follow it.
- Read dependency artifacts for context before creating new ones.
- Use `template` as the structure for your output file — fill in its sections.
- **IMPORTANT**: `context` and `rules` are constraints for YOU, not content for the file.
  - Do NOT copy `<context>`, `<rules>`, `<project_context>` blocks into the artifact.
  - These guide what you write, but should never appear in the output.

**Beads Task Guidelines**

- Title: imperative, ≤ ~70 chars (e.g., "Add JWT validator middleware").
- Description: a couple of sentences on *why* + *what* + acceptance criteria. Reference the change's proposal/design where helpful.
- Always include `--label="openspec:<name>"` so `/opsx:apply` can find them.
- Use `bd dep add` to encode ordering — apply will resume in dependency order.

**Guardrails**
- Create ALL prose artifacts needed for implementation (as defined by schema's `apply.requires`), except `tasks` — that's replaced by beads + a stub.
- Always read dependency artifacts before creating a new one.
- If context is critically unclear, ask the user — but prefer making reasonable decisions to keep momentum.
- If a change with that name already exists, ask if user wants to continue it or create a new one.
- Verify each artifact file exists after writing before proceeding to the next.
- Verify beads tasks were created (re-run `bd list --label="openspec:<name>"`) before declaring done.
